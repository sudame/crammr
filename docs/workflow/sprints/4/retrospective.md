## Keep(続けたいこと)

- **Sprint Review が機能した**: 触って初めて分かった「問題管理は要件外」「情報設計が機能を並べただけ」という気づきを、Sprint 内で吸い上げ Backlog に即反映できた。これは Review の正しい働き方の証左

- **連続 PR の段取り**: 親マージ → 子 rebase + force-push が定着(#30/#31 と同様に #36/#37 でも機能)
- **PR preview を実運用で活用**: PO が preview で実際に触って気づきを得る、というループが Sprint 1 Retro 要望の本意通り回り始めた
- **GitHub sub-issue の活用継続**: 進捗の可視化が PO 側で機能している

## Problem(うまくいかなかったこと)

- **作業が物理的にバッティング**: メイン Claude + サブエージェントが同じ working tree を触り、UX ファイルが消えたり stash と取り違えたり。解決はできたが時間ロス(数分)
- **トップ画面の情報設計が「機能を並べるだけ」だった**: 「毎回 vs 稀」の階層が無いまま積み上げ、結果 PO 指摘で全面見直しに(本 sprint 内で解消したが要件発見が遅れた構造的問題)
- **sub-issue の close 漏れ**: #25/#26 は commit で auto-close、#27/#28 は手動 close。ライフサイクルの仕舞いを忘れがち
- **#9 の見積もり粒度ミス**: 13 SP は Problem ありきの設計だった。Attempt 集計に変えたら 5 SP に縮小 → 「設計前提」が違うと SP は当てにならない

## Try(次スプリント以降で試す)

- **git worktree を既定の作業単位にする**: 並列ブランチ作業はすべて worktree で物理的に分離。サブエージェントは `isolation: "worktree"` を既定で指定。自分自身も大きいブランチ切替時は新規 worktree を切る(共有 working tree でブランチ往復しない)
- **PBI 着手前に「PO の利用シーン」を 1 行確認**: 「あなたはこの機能をいつ・どう使う?」を Issue にコメントで残してから実装。問題管理の手戻りを減らす
- **情報設計の「毎回 vs 稀」を UI 系 PBI の DoD に含める**: 新規 UI 改修時に階層を明示する(memory 化済)
- **sub-issue ライフサイクルを着手前にチェック**: PBI 着手前に親 Issue の sub-issue 一覧を確認、関連付け・close 状態を更新
- **見積もりに「設計前提」を 1 行残す**: Issue 本文に SP の根拠を簡潔に書く(再見積もり時に比較可能に)

## Sprint Review で受領した PO 観点

- **Keep**: PR preview めちゃ役立つ、機能(プラン切替・ノルマ・演習・UX)も問題なし
- **Problem(本質)**: 問題登録が面倒すぎる、自分は外部問題を解いた成果を記録したい / プラン作成が大きすぎる、毎回開いた瞬間にできることを優先する情報設計に
- **指針**:
  - プロダクトの単一情報源は Product Backlog(memory は原則・運用のみ)
  - `.handoff/` は初期叩き台、以降更新しない
  - スコープ転換は Sprint Review が機能している証左、重く受け取らない
  - 並列作業のバッティング改善のため git worktree を基本にする
